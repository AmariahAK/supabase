A supplemental script (`update.sh`) pulls a newer version of the self-hosted Supabase **configuration** on top of your existing deployment. It uses a three-way merge, so your secrets, overrides, and local edits survive, and genuine conflicts are surfaced rather than silently overwritten.

## How it works

`update.sh` compares three versions of every vendor file:

- **base** - the release your deployment is currently on, recorded in `.supabase-version` (written by `setup.sh`, then advanced to the new release after each successful update)
- **new** - the release you are updating to (the latest `self-hosted/v*` tag by default)
- **user's** - the files in your deployment directory

It applies the changes between base and new to your files. Where you never edited a file, it updates cleanly. Where you edited a file but the release did not touch the same lines, your edit is kept. Only when both changed the same lines do you get a conflict to resolve.

Your secrets and data are never merged. The values in the `.env` file you set are kept (new keys from `.env.example` are appended), and paths listed in `.gitignore` are left untouched.

## Before you start

<Admonition type="danger" title="Back up your database">

`update.sh` backs up your configuration files to `backups/`, but it does not back up Postgres or Storage data. Back those up separately first.

</Admonition>

- Run from your deployment directory (where `docker-compose.yml` and `.env` live).
- `update.sh` needs `git` and `jq` on the host (both are installed by `setup.sh`).
- Deployments created with `setup.sh` record their version in `.supabase-version`, and `update.sh` advances it after each successful update. If that file is missing (older or manual installs), refer to [Setting a recorded version](#setting-a-recorded-version).

## How to update

Preview what would change, without affecting anything:

```sh
sh update.sh --dry-run
```

Apply the update (targets the latest `self-hosted/v*` release):

```sh
sh update.sh
```

Pull the new images and recreate the containers:

```sh
sh run.sh pull
sh run.sh recreate
```

Review the summary `update.sh` prints - especially any conflicts, new `.env` keys, and breaking-change notices.

### What it changes

Updated:

- `docker-compose.yml`, override templates, `volumes/*`, scripts, `.env.example`
- `volumes/functions/main/index.ts` (vendor bundle)

Excluded:

- Current `.env` configuration
- Paths in `.gitignore`: data directories, snippets, your edge functions

## Breaking changes

Some releases need a manual step before their files can be applied - for example a Postgres major upgrade. `update.sh` detects these from the release manifest and stops to list them **before** modifying anything, so you can abort with your deployment untouched.

Complete the listed steps (including any `utils/*.sh` migration script mentioned), then re-run `sh update.sh`. Use `--yes` only when you have already done the required work.

## Resolving conflicts

If the summary lists conflicts, `update.sh` has written standard merge markers into those files and exits with status `2`. Open each file, pick the correct content, and remove the `<<<<<<<`, `=======`, and `>>>>>>>` markers:

```yaml
<<<<<<< yours (docker-compose.yml)
      image: supabase/studio:your-pinned-tag
=======
      image: supabase/studio:new-tag
>>>>>>> new (self-hosted/v0.7.0)
```

Then start the stack:

```sh
sh run.sh pull
sh run.sh recreate
```

A conflict means you edited a vendor file and the release changed the same lines. If you enabled auth providers by uncommenting lines in `docker-compose.yml`, this is where you re-apply them when a release reworks those lines.

While conflicts remain, `update.sh` does **not** advance `.supabase-version` — it records the new release only on a clean run. After you resolve the markers, re-run `sh update.sh` to finalize the version stamp.

## Update to a specific release

```sh
sh update.sh --to self-hosted/v0.7.0
```

See the [changelog](https://github.com/supabase/supabase/blob/master/docker/CHANGELOG.md) for available releases.

## Setting a recorded version

Without `.supabase-version`, `update.sh` cannot merge safely and runs in **report-only** mode: it lists what is new but writes nothing. To fix it once, identify the release your deployment came from (compare the image tags in `docker-compose.yml` and `.env` against [versions.md](https://github.com/supabase/supabase/blob/master/docker/versions.md)) and record it:

```sh
printf 'ref=self-hosted/v0.7.0\n' > .supabase-version
```

Or pass it for a single run:

```sh
sh update.sh --from self-hosted/v0.7.0
```

## Restore from backup

**Configuration backups** are written to `backups/pre-update-*.tgz` before each update. If something goes wrong, extract or compare against that archive.

## Command-line options

| Flag           | Purpose                                                     |
| -------------- | ----------------------------------------------------------- |
| `--dry-run`    | Show the plan; write nothing                                |
| `--to <tag>`   | Update to a specific release                                |
| `--from <ref>` | Supply the base version when `.supabase-version` is missing |
| `--yes`        | Skip the breaking-change confirmation prompt                |
